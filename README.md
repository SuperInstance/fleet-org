# 🏗️ Fleet Org Chart — Agent Middle Management

Oracle1 is Managing Director. Below are specialized repo-agents that own domains.
Each is a GitHub repo with BOOTCAMP.md, CHARTER.md, and bottle system.

## Hierarchy

```
Captain Casey 🎣
  └── Oracle1 🔮 — Managing Director (coordination, architecture, strategy)
        ├── Mechanic 🔧 — Fleet Maintenance (repo health, fixes, CI)
        ├── Architect 📐 — CTO (ISA design, runtime architecture, spec review)
        ├── Foreman 🏗️ — Project Manager (task board, priorities, blockers)
        ├── Auditor 📊 — Quality Lead (conformance, benchmarks, regression)
        ├── Quartermaster 📦 — Release Manager (versioning, changelogs, packaging)
        └── [Grow as needed from task patterns]
```

## Spawning Protocol

1. Identify a domain that keeps appearing in tasks
2. Create repo with CHARTER.md (what they own) + BOOTCAMP.md (how to replace them)
3. Assign initial tasks from the task board
4. Agent self-specializes through work — we don't pick roles, the work does
5. If an agent's expertise drifts, let it. Park the old rigging, pick up new.

## What Makes a Good Repo-Agent

- **Narrow charter** — owns one domain deeply, not everything shallowly
- **Bootcamp** — any model can become them by cloning and reading
- **Bottle-aware** — checks messages, responds to fleet needs
- **Self-documenting** — every decision written down in the repo
- **Disposable** — if a better agent comes along, the repo survives the swap

## Spawning Signals

When you see these patterns, it's time to spawn a new repo-agent:

- Same category of task appearing 3+ times on the board
- Oracle1 spending >20% of time on one domain
- An agent developing expertise that deserves its own vessel
- A knowledge gap that no agent can fill yet — create the role, let work fill it
